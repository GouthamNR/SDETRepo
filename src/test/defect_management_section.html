üéØ Defect Management Overview

Understanding bugs and defects in software testing

üîß What is a Defect?

Definition: A defect (or bug) is a deviation from expected behavior where the software does not meet its requirements.

Examples:

  • ‚úó Login field accepts 12 characters (requirement: only 10)
  • ‚úó "Save" button doesn't save data
  • ‚úó Amount transfer allows transfer > account balance
  • ‚úó Error message is in wrong language

‚ùì Why Do Software Defects Exist?

üó£Ô∏è Miscommunication

Unclear or missing requirements between customer and dev team

‚öôÔ∏è Complexity

Complex logic makes it difficult to handle all edge cases

üíª Programming Errors

Developer mistakes, typos, wrong logic

üìù Changing Requirements

Requirements change mid-development, causing misalignment

‚è∞ Time Pressure

Rushing to meet deadlines causes shortcuts and bugs

üîß Technical Debt

Using old libraries, unsupported technologies

üí° The Quality Principle

üéì Fundamental Testing Principle:

"There is NO product with zero defects!"


Why?
  • Software is infinitely complex
  • Unlimited test scenarios possible
  • Always some edge cases not covered
  • New features may introduce new bugs

Implication:

Goal is not 100% bug-free, but Acceptable Quality Level (AQL) - managing defects effectively

üî¥ SEVERITY Levels (4 Types)

üö´ BLOCKER / SHOWSTOPPER

Definition:

Testing cannot proceed. Major flow completely broken.

Examples:

  • Login doesn't work at all ‚Üí Can't access app
  • Database connection broken ‚Üí No data retrieval
  • Application crashes on startup
  • Payment processing completely fails

Impact:

üî¥ CRITICAL - Must fix immediately

üî¥ CRITICAL

Definition:

Large piece of functionality or major system component completely broken. Business affected significantly.

Examples:

  • Amount transfer allows transfer > balance
  • Email sending doesn't work
  • Report generation fails
  • Core business transaction broken

Impact:

Loss to customer = HIGH

üü† MAJOR

Definition:

Important feature not working properly. Workaround exists. Testing continues.

Examples:

  • UI elements misaligned
  • Font color wrong
  • Button in wrong position
  • Sorting doesn't work (but data is viewable)

Impact:

Look & feel affected, functionality works

üü° MINOR

Definition:

Minor issue. Limited functionality loss. Acceptable workaround available. Testing proceeds normally.

Examples:

  • Typo in label or error message
  • Help text is incorrect
  • Tooltip missing
  • Cosmetic appearance issue

Impact:

Low - Minimal business impact

üéØ How to Assign Severity?

Scenario Severity Assignment Reason
Core feature (Login) broken üî¥ CRITICAL Core features are critical
Non-core feature (Help) broken üü° MINOR Non-core features are minor
Financial transaction fails üö´ BLOCKER Business revenue affected
Cosmetic issue (color wrong) üü† MAJOR Look & feel affected
Spelling mistake in message üü° MINOR Cosmetic only, no functionality loss

‚è±Ô∏è PRIORITY Levels (4 Types)

P1 - HIGH PRIORITY

Definition: Must fix immediately. Major impact on customer.

When assigned: Severe functionality broken or business revenue affected

Fix deadline: Same day / Next build

P2 - MEDIUM PRIORITY

Definition: Should fix before release. Moderate customer impact.

When assigned: Important feature not working properly

Fix deadline: Within 2-3 days

P3 - LOW PRIORITY

Definition: Fix if time permits. Minor customer impact.

When assigned: Minor issues, cosmetic problems

Fix deadline: Future release (can defer)

P4 - TRIVIAL

Definition: Minimal impact. Can be ignored.

When assigned: Very minor cosmetic issues

Fix deadline: No specific deadline, skip if necessary

üí° Important Notes

  • ‚úÖ Severity is objective: Based on impact, facts
  • ‚úÖ Priority is subjective: Based on business decision
  • ‚úÖ Same bug can have different severity/priority:
    • Typo on internal tool = MINOR / P3
    • Same typo on customer-facing website = MAJOR / P1
  • ‚úÖ Developer fixes by priority (not severity)
    ‚Üí P1 defects fixed first, regardless of severity
  • ‚úÖ Test team assigns severity, not developers

üîÅ Avoiding Duplicate Bug Reports

‚ùì Why Duplicates Happen?

  • Multiple testers testing same feature
  • Same bug in different modules (common features)
  • Testers don't search existing defects

‚úÖ How to Avoid Duplicates?

BEFORE reporting:

  1. Click "Advanced Search" in tool
  2. Enter keywords (e.g., "password", "login")
  3. Search the defect repository
  4. Review existing bugs with similar keywords
  5. Read details carefully - might be different bug!
  6. If NOT a duplicate ‚Üí Report new bug
  7. If YES duplicate ‚Üí Link to original bug

üìö Example: Duplicate Bug Detection

Scenario: Two testers find similar bugs


Tester A Finds: "Password field accepts 15 characters (requirement: 10)"
‚Üí Reports: BUG-1234

‚Üí Searches for "password" ‚Üí Finds BUG-1234
‚Üí Reads BUG-1234 details ‚Üí DIFFERENT BUG!
‚Üí Reports: BUG-1235 (separate bug)


Why different?
BUG-1234: Character count validation issue
BUG-1235: Character type validation issue
Both are bugs, but different root causes

üìä Defect Statuses & Actions

Understanding why defects are rejected or postponed

‚ùå REJECT: Why Defects Are Rejected

1Ô∏è‚É£ Misunderstanding of Requirements

Situation: Tester thought behavior was wrong, but it's actually as per requirements

Example:

Tester reports: "Search field only accepts 10 characters"
Dev Lead: "That's correct per requirement v2.3 - max 10 chars"
‚Üí REJECTED: Not a bug, expected behavior

Resolution: Tester should review updated requirements

2Ô∏è‚É£ Wrong Installation/Configuration

Situation: Tester installed or configured product incorrectly

Example:

Tester reports: "Database connection fails"
Dev Lead checks: "Database credentials wrong in your config file"
‚Üí REJECTED: User error, not a bug

Resolution: Reconfigure and verify settings

3Ô∏è‚É£ Referring to Old Requirements

Situation: Requirements changed, but tester is using old version

Example:

Old Requirement (v1.0): "Show Sales link on homepage"
New Requirement (v2.0): "Remove Sales link (customer requested)"
Tester using v1.0 reports: "Sales link is missing!"
‚Üí REJECTED: Refer to updated requirements v2.0

Resolution: Use latest requirements document

4Ô∏è‚É£ Extra Feature (Not in Scope)

Situation: Developer added extra feature, not in requirements

Example:

Tester reports: "No CLEAR button on form (inconvenient)"
Dev Lead: "CLEAR button not in requirements"
‚Üí REJECTED: Not a bug, it's a feature enhancement request (RFE)

Resolution: Converted to RFE (Request for Enhancement)

üîç NOT REPRODUCIBLE: Inconsistent Bugs

Situation: Developer cannot reproduce the bug, but tester reports it happening consistently

Reason 1: Incomplete Steps

Tester Reports: "Sent mails not appearing in Sent Items"
Steps: "Open app and check Sent Items"

Problem: Missing crucial steps!

Complete Steps:
1. Login to specific mailbox with 100+ existing mails
2. Compose and send new mail
3. Immediately check Sent Items
‚Üí New mail not visible (bug occurs)

Solution: Provide step-by-step reproduction including delays/wait times

Reason 2: Build Mismatch

Scenario: Tester reports bug from Build 1, but developer is testing Build 2

Build 1: Has bugs X and Y

Build 2: Bug X fixed, Bug Y still exists

Problem: Developer testing Build 2 cannot find Bug X

Solution: Specify exact build number in defect report

Reason 3: Inconsistent/Intermittent Bug

Scenario: Bug happens sometimes, not always (race condition)

Tester's Experience:
1st attempt: Compose mail ‚Üí Send ‚Üí Mail appears in Sent Items ‚úì
2nd attempt: Compose mail ‚Üí Send ‚Üí Mail NOT in Sent Items ‚úó
3rd attempt: Works ‚úì

Problem: Inconsistent behavior, hard to reproduce

Solution: Mention "intermittent" in report, provide timing details

  • ‚ùå Using Java - JavaScript functionality not available

‚è∏Ô∏è POSTPONED & DEFERRED: Fixing in Future Release

POSTPONED (In Current Release)

When: Bug found near end of release (last 2-3 days)

Why: No time to fix and retest before release date

Decision: Fix in next release instead

Status: POSTPONED (open, not fixed)

DEFERRED (To Specific Future Release)

When: Bug is known and acknowledged but low priority

Why: Business decided not worth fixing now

Decision: Fix in v2.5 or future release

Status: DEFERRED (scheduled for future)

WON'T FIX

When: Bug acknowledged but won't be fixed

Why: Very low priority, no business value, edge case

Decision: Permanent decision not to fix

Status: WON'T FIX (closed without fixing)

üí° RFE: Request For Enhancement

Definition: Not a bug, but a feature request. Developers acknowledge it's good to have but not in original requirements.

Example Scenario:

1. Tester Reports: "No CLEAR button on form"

2. Developer Sees: "CLEAR button not in requirements"

3. Developer Decision: "It's a good idea, but not originally specified"

4. Status Changed to: RFE (Request For Enhancement)

5. Business Impact: Can charge customer for this feature!

üí∞ Why RFE matters?

RFE = Extra work ‚Üí Can bill customer

Bug = Expected behavior ‚Üí No extra charge

üìà Defect Metrics & Analysis

üìä Key Defect Metrics

1. Defect Density

Formula: Total Defects / 1000 Lines of Code (KLOC)

Industry Benchmark: 40-60 defects per KLOC

Interpretation: How many bugs per 1000 lines?

2. Defect Detection Rate (DDR)

Formula: (Defects found during testing / Total defects) √ó 100

Goal: 95-98%

Interpretation: How effective is testing at finding bugs?

3. Defect Escape Rate

Formula: (Defects found after release / Total defects) √ó 100

Acceptable: 2-5%

Interpretation: How many bugs escape to production?

4. Fix Rate

Formula: (Defects fixed / Defects reported) √ó 100

Target: 60-90% during testing cycle

Interpretation: What percentage of reported bugs get fixed?

5. Bug Aging

Calculation: Time from OPEN to CLOSED

Average: 5-15 days

Interpretation: How long does a bug take to resolve?

6. Pass Rate

Formula: (Executed test cases PASSED / Total executed) √ó 100

Benchmark: 85%+ for release quality

Interpretation: What percentage of tests pass?
