Understanding why and when to write test cases

üéØ What is a Test Case?

Definition: A test case is a set of input values, execution preconditions, expected results, and post-execution conditions designed to cover a particular test objective.

Key Purpose: To ensure consistent, quality testing regardless of who executes the test

‚úÖ Better Coverage

Document all possible scenarios systematically

‚úÖ Consistency

Testing doesn't depend on tester's mood or experience

‚úÖ Knowledge Transfer

New testers can quickly understand what to test

‚úÖ Process-Based

Depend on procedures, not on individual testers

‚úÖ Traceability

Link requirements to test cases to automation scripts

‚úÖ Reusability

Use same test cases across multiple releases

üéØ Test Case Design Techniques

Methods to systematically derive effective test cases

1Ô∏è‚É£ Error Guessing

üîç Definition:

Based on experience, guess what errors might exist and test for them

üìã How It Works:

Rule: If field accepts only positive integers, test with:

  • ‚ùå Decimal numbers: 100.50
  • ‚ùå Negative numbers: -100
  • ‚ùå Special characters: 100@$
  • ‚ùå Text: "hundred"
  • ‚ùå Blank/spaces
  • ‚úÖ Valid positive integers: 100, 5000

‚ö° Advantages:

  • Quick and intuitive
  • Catches obvious bugs
  • No formal procedures needed

‚ùå Disadvantages:

  • Depends on tester experience
  • Miss boundary cases
  • Inconsistent across team
  • May have gaps in coverage

üéØ When to Use:

Alongside other techniques, for exploratory testing, when requirements are unclear

2Ô∏è‚É£ Equivalence Partitioning (EP)

üîç Definition:

Divide input domain into equivalent groups and test one from each group

üìã Pressman's Rule:

For Range Input:
  • ‚úÖ 1 Valid value (inside range)
  • ‚ùå 2 Invalid values (below & above range)

Example (Amount: 100-5000):
  • ‚úÖ Valid: 2000 (inside 100-5000)
  • ‚ùå Invalid: 50 (below 100)
  • ‚ùå Invalid: 6000 (above 5000)

üé® Visual Representation:

Invalid
< 100
Valid
100-5000
Invalid
> 5000

‚úÖ Advantages:

  • Systematic approach
  • Good coverage with few test cases
  • Reduces redundant test cases
  • Easy to understand and implement

‚è±Ô∏è Effort: 3 test cases for range field

3Ô∏è‚É£ Boundary Value Analysis (BVA)

üîç Definition:

Test values at boundaries where behavior might change

üìã BVA Rule (For Range A-B):

Test: A-1, A, A+1, B-1, B, B+1

üé® Example (Amount: 100-5000):

Lower Boundary:
  • 99 (A-1)
  • 100 (A)
  • 101 (A+1)
Upper Boundary:
  • 4999 (B-1)
  • 5000 (B)
  • 5001 (B+1)

üí° Why BVA Works:

Developers often make mistakes at boundaries (e.g., using < instead of <=)

Common Bug Pattern:
If (amount < 100) Error // Should be: <= 99
If (amount > 5000) Error // Should be: >= 5001

üÜö BVA vs EP:

BVA: More test cases (6) but catches subtle boundary bugs
EP: Fewer test cases (3) but may miss boundary issues
Recommendation: Use BVA when boundary behavior matters

‚è±Ô∏è Effort: 6 test cases for range field

4Ô∏è‚É£ Practice Method (Enhanced Equivalence)

üîç When Used:

When simple EP doesn't catch business logic variations

üìã Example: Insurance Premium Policy

Business Rules:

  • Age < 10: Not eligible
  • Age 10-25: 2% premium deduction
  • Age 26-55: 3% premium deduction
  • Age > 55: 5% premium deduction

‚ùå Simple EP (3 cases insufficient):

  • Valid: 20 (inside 0-100) ‚Üí Misses different deductions!
  • Invalid: -5, 150

‚úÖ Practice Method (8 cases):

Age Value Category Expected Behavior
5 < 10 Reject
10 10-25 (start) 2% deduction
17 10-25 (middle) 2% deduction
25 10-25 (end) 2% deduction
26 26-55 (start) 3% deduction
40 26-55 (middle) 3% deduction
55 26-55 (end) 3% deduction
56 > 55 5% deduction

üí° Key Insight:

Practice method finds boundary transitions where business logic changes

üéØ When to Use:

  • Complex business logic with multiple ranges
  • Policy-based systems
  • Insurance, banking, taxation calculations

‚è±Ô∏è Effort: 8+ test cases per field

üìä Design Techniques Comparison

‚úçÔ∏è Test Case Writing Best Practices

Professional approaches to high-quality test case creation

‚úÖ Be Specific & Clear

‚ùå Wrong: "Enter data and verify"

‚úÖ Right: "Enter amount '12000' in Amount field and verify error message 'Cannot exceed balance' displays"

‚úÖ Use Expected Result Format

Say "Should be" or "Must be" instead of hardcoding

‚ùå Wrong: "Error displayed"

‚úÖ Right: "Error message 'Invalid input' must be displayed"

‚úÖ Focus on Critical Steps

Don't elaborate on every single action. Focus on steps directly related to the test objective.

Example: Skip "Open browser, load page, login..." if not relevant to Amount Transfer validation

‚úÖ Highlight Object Names

Make UI elements clear for better identification

‚úÖ "Click the 'Submit' button (green button at bottom-right)"

‚úÖ Write Generic Test Cases

Don't hardcode values that change:

‚ùå Wrong: "Enter hostname: 192.168.1.1"

‚úÖ Right: "Enter hostname: [From Test Data - Environment Config]"

Reason: Hostname changes in different environments (Dev, QA, Prod)

‚úÖ Optimize for Execution Speed

Organize steps so related actions are grouped together

This reduces navigation time during execution

‚úÖ Include Navigation Steps

Always start with navigation steps for clarity

‚úÖ "1. Navigate to CBO homepage
2. Click Login
3. Navigate to Amount Transfer
4. [Then specific test steps]"

‚úÖ Apply Design Techniques

Use BVA, EP, or Practice Method to derive comprehensive test values

Don't guess - systematically derive test data

‚ö†Ô∏è Common Test Case Writing Mistakes

‚ùå Mistake 1: Writing Like Documentation

Wrong Approach:

"The user opens the application and navigates through the beautiful interface to find the amount transfer feature which is located on the left menu. He then carefully fills in all the required fields..."

Correct Approach:

"1. Navigate to Amount Transfer feature
2. Enter amount: 12000
3. Click Submit"

‚ùå Mistake 2: No Expected Results

Wrong: Just steps, no "Expected Result" column

Correct: Every step has explicit expected result for validation

‚ùå Mistake 3: Mixing FT, IT, ST in One Case

Wrong: "Test Login ‚Üí Test Database Connection ‚Üí Test Entire Workflow"

Correct: Separate test cases for each level
‚Ä¢ FT: Test Login feature alone
‚Ä¢ IT: Test Login + Amount Transfer interaction
‚Ä¢ ST: Test complete end-to-end workflow

‚ùå Mistake 4: Hardcoding Environment-Specific Data

Wrong: "URL: 192.168.1.1:8080/cbo"

Correct: "[From Test Data] - See Environment Config file"

Reason: URL changes across environments (Dev/QA/Prod)

‚ùå Mistake 5: Too Many Steps

Wrong: 100 steps in single test case

Correct: 15-20 steps maximum, break into separate cases if needed

Why: Easier to execute, identify failures, and troubleshoot

‚öôÔ∏è Test Case Execution & Reporting

Running test cases and documenting results

üîÑ Test Execution Process

1Ô∏è‚É£ Preparation (Before Execution)

  • Retrieve approved test cases from repository
  • Create fresh test data (clean environment)
  • Set up test environment (db, servers, config)
  • Verify build/application is deployed and ready

2Ô∏è‚É£ Execution (Running Tests)

  • Execute test cases step-by-step following documented steps
  • Document actual results for each step
  • Compare Expected vs. Actual results
  • If mismatch: FAIL ‚Üí Log defect
  • If match: PASS ‚Üí Continue to next step

3Ô∏è‚É£ Defect Logging (On Failures)

  • Create defect report with step number that failed
  • Attach screenshots/logs as evidence
  • Assign to developer for fixing

4Ô∏è‚É£ Reporting (After Execution Cycle)

  • Create Test Execution Report (TER)
  • Calculate metrics (Pass %, Fail %)
  • Send to stakeholders

üìä Test Case Effectiveness Metrics

Measure: How effective are your test cases at catching bugs?

Key Metrics:

  • Bug Detection Rate (BDR): Bugs found by test case / Total bugs in feature
  • Test Case Efficiency: Bugs found / Number of test cases
  • Code Coverage: Lines of code covered by tests
  • False Positive Rate: Defects reported as bugs but are actually features

Example:

  • Test case found 5 bugs from 8 total bugs ‚Üí BDR = 62.5%
  • 10 test cases found 5 bugs ‚Üí Efficiency = 0.5 bugs/test case
  • Low metrics indicate test cases need improvement

Regression Test Selection Methods:

  • Risk-Based: Select high-risk test cases only
  • Change-Based: Select test cases affected by code changes
  • Hybrid: Combine methods (10 cycles change-based + 1 cycle all tests)
