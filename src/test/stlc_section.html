üíß Waterfall Model

Requirements
Analysis
Design
Coding

Characteristics:

  • Sequential, phase-by-phase approach
  • Testing starts AFTER development is complete
  • No backtracking allowed
  • Rigid structure, fixed requirements

‚úÖ Advantages:

  • Simple and easy to understand
  • Well-organized with clear deliverables
  • Stable for projects with fixed requirements

‚ùå Disadvantages:

  • Late defect detection (expensive to fix)
  • Bugs flow downward through phases
  • No flexibility for requirement changes
  • High risk for complex projects

üéØ When to Use:

Short-term projects, well-defined requirements, simple applications (e.g., calculators)

‚ö†Ô∏è Testing Challenges:

High Risk: Testing only validates if requirements are met, not if they're right

üåÄ Spiral Model (Iterative)

Combines advantages of waterfall and iterative approaches. Development and testing happen in cycles for each module.

Cycle 1: Module A (Req ‚Üí Design ‚Üí Code ‚Üí Test)
Cycle 2: Module B (Test B + Integration B-A)
Cycle 3: Module C (Test C + Integration C-A-B)

Testing Strategy:

  • Unit test each module after coding
  • Integration test with previously tested modules
  • Continuous testing in each cycle

‚úÖ Advantages:

  • Early defect detection
  • Allows requirement changes between cycles
  • Lower cost of change
  • Risk identification at each cycle

üéØ Best For:

Projects with module dependencies, staged requirements, large applications (MS-Excel style)

V-Model (Verification & Validation)

Key Innovation: Testing starts from the requirement stage, not after coding

CRS ‚Üí Review CRS, Write ATP
SRS ‚Üí Review SRS, Write STP
HLD ‚Üí Review HLD, Write ITP
LLD ‚Üí Review LLD, Write FTP
CODING
‚Üê Unit Testing ‚Üê LLD Review
‚Üê Functional Testing ‚Üê HLD Review
‚Üê Integration Testing ‚Üê SRS Review
‚Üê System Testing ‚Üê CRS Review

‚úÖ Key Advantages:

  • Early Testing: Catches defects at early stages
  • Parallel Activities: Dev and Testing teams work simultaneously
  • Reduced Rework: Avoids downward flow of defects
  • Lower Cost: Less re-work needed
  • Higher Quality: Testing involved at every stage

‚ùå Disadvantages:

  • High initial investment (testing team needed from start)
  • Extensive documentation requirements
  • No working prototype until late in cycle

üéØ When to Use:

Long-term projects, complex applications, quality-critical systems, regulated industries

üìà Advanced Quality Metrics

1. Defect Removal Efficiency (DRE):

DRE = (Pre-delivery Defects Found) / (Total Defects = Pre + Post-delivery) √ó 100

Goal: 95%+ (Catch defects before production)

2. Defect Density:

Defects Found / Size (KLOC)

Goal: < 0.5 defects per KLOC

3. Test Case Effectiveness:

= (Defects Found by Test Case) / (Total Test Cases Run) √ó 100

Goal: High effectiveness means good quality tests

4. Cumulative Weighted Defect Density:

= (Critical√ó4 + Major√ó3 + Minor√ó2 + Trivial√ó1) / Size (KLOC)

Goal: Track quality trend over releases

üîÑ Advanced Regression Testing

Strategic approach to managing regression across releases

Unit Regression Testing:

Test only the bug fix/change - ensure fix doesn't break immediate area

Regional Regression Testing:

Test the change + impact areas - identify side effects

Preferred method - balances coverage with efficiency

Full Regression Testing:

Test all changes + entire product - comprehensive coverage

Strategy: Do 10 cycles RRT, 1 cycle FRT, repeat pattern

üíæ Advanced Test Data Management

  • Production-like Data: Use sanitized production data for realistic testing
  • Data Masking: Mask sensitive information (PII, financial data)
  • Data Refresh: Reset data between test cycles
  • Boundary Values: Test min/max values and edge cases
  • Negative Test Data: Invalid formats, null values, special characters
  • Large Dataset Testing: Test with high volume (stress testing data)

üéØ Key Takeaways from STLC & Process

  • 1. STLC Matters: Systematic approach ensures quality and reduces risks
  • 2. Model Selection: Choose SDLC model based on project needs (Waterfall, Spiral, Agile)
  • 3. Early Testing: Test from requirement stage (V-Model) vs. after coding (Waterfall)
  • 4. Entry/Exit Criteria: Clear gates prevent moving forward with quality issues
  • 5. Continuous Improvement: Retrospectives and metrics drive process excellence
  • 6. Risk-Based Approach: Focus testing on high-impact, high-probability areas
  • 7. Agile Adaptation: STLC applies to Agile too - adapt iterations for sprints
  • 8. Metrics Drive Decisions: Use DRE, defect density, coverage to guide testing
