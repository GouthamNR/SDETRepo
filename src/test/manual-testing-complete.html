What is Manual Testing?

Manual testing is the process of testing software applications by manually executing test cases without using any automation tools. A QA engineer performs all test activities manually, simulating real-world user behavior and scenarios.

üí° Key Point: Manual testing requires human intelligence, experience, and intuition to find defects that automated tests might miss.

Key Responsibilities of Manual Testers

üîç Test Design

Create comprehensive test cases covering all scenarios and edge cases

‚ñ∂Ô∏è Test Execution

Execute test cases step by step and verify expected results

üêõ Defect Identification

Identify and document bugs with detailed reproduction steps

üìä Analysis

Analyze defects and validate fixes through retesting

Why Manual Testing Still Matters

  • User Experience Testing: Only humans can truly evaluate UX/UI quality
  • Exploratory Testing: Finding unexpected bugs requires human intuition
  • Complex Scenarios: Real-world workflows often too complex to automate
  • Ad-hoc Testing: Quick validation without predefined test cases
  • New Features: Testing fresh features before automation scripts exist

Test Case Writing Best Practices

‚úì DO:
  • Use clear, specific language that anyone can understand
  • Test one thing per test case (single responsibility)
  • Include both positive and negative scenarios
  • Make test cases independent and reusable
  • Use actual data values instead of "any data"
‚úó DON'T:
  • Create vague test cases like "test login"
  • Include multiple assertions in one test case
  • Forget to include expected results
  • Leave test data to tester imagination
  • Skip boundary value testing

Types of Test Cases

Positive Test Cases

Test with valid data and valid scenarios ‚Üí System should work correctly

Example: Login with correct username and password

Negative Test Cases

Test with invalid data or error scenarios ‚Üí System should handle gracefully

Example: Login with incorrect password, empty fields, SQL injection attempts

Boundary Value Test Cases

Test at limits of input ranges ‚Üí Test extreme values

Example: Age field testing with 0, 1, 120, 999, -1 values

‚ñ∂Ô∏è Test Execution & Best Practices

Step-by-Step Test Execution Process

  1. Preparation: Verify test environment is ready, test data is available
  2. Pre-Execution Checks: Ensure application launches, basic functionality works
  3. Execute Step 1: Perform first test action, note actual result
  4. Compare with Expected: Does actual match expected result?
  5. Document Status: Mark as PASS, FAIL, or BLOCKED
  6. Continue Steps: Repeat for all remaining test steps
  7. Log Defects: If any step fails, create detailed bug report
  8. Update Test Case: Record actual results and test date/tester name

Best Practices During Execution

üéØ Focus & Concentration

Pay close attention to details, every pixel matters in UI testing

üìù Document Everything

Take screenshots, note error messages, record exact steps to reproduce

üê¢ Go Slow

Don't rush testing, follow each step exactly as written

üîÑ Retesting

Always verify bug fixes by retesting the same test case

Test Coverage Considerations

Coverage Goals:
  • Functional Coverage: All features have at least one test case
  • Risk Coverage: Critical features have maximum test cases
  • Data Coverage: All data types and ranges tested (valid, invalid, boundary)
  • Path Coverage: All user workflows and decision paths tested
  • Error Coverage: Error handling and edge cases covered

Bug Reporting Best Practices

‚úì DO:
  • Be specific and concise in bug title
  • Include exact reproduction steps with actual data
  • Attach relevant screenshots or videos
  • Include environment details (OS, browser, version, build)
  • Test your reproduction steps before reporting
‚úó DON'T:
  • Report vague bugs like "system is broken"
  • Mix multiple bugs in one report
  • Forget to mention when/how you found the bug
  • Set wrong severity/priority (let developer decide)
  • Assume developers know what you're talking about

Software testing is divided into two main categories that work together to ensure comprehensive quality assurance:

Functional vs Non-Functional Testing Overview

üîç Functional Testing

Tests WHAT the software does - verifies features work according to requirements

  • ‚úì Requirement-focused
  • ‚úì Business logic validation
  • ‚úì Positive & negative testing
  • ‚úì End-user scenarios
‚öôÔ∏è Non-Functional Testing

Tests HOW the software works - verifies performance, security, reliability

  • ‚úì Quality attributes
  • ‚úì Performance metrics
  • ‚úì System reliability
  • ‚úì User experience

Functional Testing Types (13+)

1. Unit Testing

Badge: Developers | White Box

What: Individual module/component testing

When: During development, before integration

Who: Developers only

Example: Test login() function with valid credentials, invalid credentials, empty fields, special characters

2. Integration Testing

Badge: Devs + QA | Gray Box

What: Testing interaction between multiple modules

When: After unit testing when modules combine

Who: Developers + QA Team

Approaches:

  • Top-Down: Test high-level modules first using stubs
  • Bottom-Up: Test low-level modules first using drivers
  • Big Bang: Combine all modules at once (not recommended)

Example: Amount Transfer: Verify debit from Account A flows correctly to credit in Account B

3. System Testing

Badge: QA Team | Black Box

What: End-to-end testing of complete integrated system

When: After integration testing passes

Who: QA Team (no code visibility)

Environment: Production-like test environment

Example: Complete Banking Loan Process - Application ‚Üí Approval ‚Üí Disbursement including interest calculation, documents, workflow

4. User Acceptance Testing (UAT)

Badge: End Users | Black Box

What: Final testing by end-users/business stakeholders

When: After system testing passes

Who: Business users, domain experts, customers

Data: Real-time business scenarios

  • Alpha Testing: At developer site by domain experts
  • Beta Testing: At customer site before live release

Example: Fed-Ex employees test payment processing with actual Fed-Ex workflows and real transaction scenarios

üî• 5. Smoke Testing (Build Verification Testing)

Badge: Quick Check | CRITICAL

What: Quick sanity check of basic/critical features

When: IMMEDIATELY when new build received

Duration: 30 minutes to 2 hours

Testing Type: POSITIVE TESTING ONLY (valid data)

Critical Checks:

  • ‚úì Application launches
  • ‚úì Login works
  • ‚úì Homepage loads
  • ‚úì Navigation accessible
  • ‚úì Key features responsive
‚ö†Ô∏è Decision: If ANY check FAILS ‚Üí REJECT BUILD immediately. Don't waste time on detailed testing!
6. Sanity Testing

Badge: Deep Testing | Critical Features

vs Smoke Testing: Narrow & Deep (vs Wide & Shallow)

Approach: Manual, unscripted (experience-based)

Focus: Very important/critical features only

Example: Banking App - thoroughly test Login, Amount Transfer, Balance Check with multiple scenarios each

‚≠ê 7. Regression Testing

Badge: Critical | Repetitive

Purpose: Ensure existing features still work after bug fixes or new features added

‚ö° Why Critical: One bug fix can break multiple other features!

Types:

  • Unit Regression: Test only changed code
  • Regional Regression: Changed code + impact areas (RECOMMENDED)
  • Full Regression: Entire application (time-consuming but thorough)
üí° Solution: Automate regression testing with QTP/Selenium/Cypress for efficient re-execution
8. Retesting vs Regression
Aspect Retesting Regression Testing
Scope Only specific bug fixed Entire application/modules
Purpose Verify specific bug is fixed Ensure fix doesn't break other features
When After each developer fix Comprehensive plan
Flow Find bug ‚Üí Fix ‚Üí Retest ‚úì Fix ‚Üí Regression check entire system
9. End-to-End Testing

Badge: Complete Workflow

What: Complete business workflow from start to finish including all systems

E-Commerce Example Workflow: Login ‚Üí Browse ‚Üí Add Cart ‚Üí Checkout ‚Üí Payment ‚Üí Confirmation

Data Used: Real-like production data through entire workflow

10. Ad-Hoc Testing (Monkey Testing)

Badge: Random | Experience-Based

What: Random testing without predefined procedures or test cases

Why Important: End-users behave unpredictably, not systematically like testers

Examples:

  • üî¥ Login ‚Üí Immediately click Back (should NOT show inbox)
  • üî¥ Copy URL ‚Üí Logout ‚Üí Access URL again (should NOT work)
  • üî¥ Rapid clicks on submit button multiple times
üí° Real Benefit: Ad-hoc often finds bugs that systematic testing misses because it mimics actual user behavior!
11. Exploratory Testing

Badge: Unscripted | Learning

What: Simultaneous test design and execution without pre-planned cases

Who: Experienced testers with product knowledge

When: Requirements unclear OR exploring application behavior

Approach: Adapts to findings in real-time, learns as testing progresses

üìù Note: Even though unscripted, always document findings for future reference

12. Mutation Testing

Badge: Quality Check | Developers

Process:

  1. Developers intentionally inject defects
  2. Testers test the application
  3. If caught ‚Üí Good test quality

Purpose: Verify tester quality and test case effectiveness

13. Exhaustive Testing

Badge: Comprehensive | Time-Intensive

What: Testing ALL positive AND negative scenarios for a feature

üî¥ Reality Check: Often impossible due to combinatorial explosion

Example: 15 input fields √ó 5 values each = 30+ billion test combinations!

Usage: Only for critical features where quality is absolutely paramount

Non-Functional Testing Types (14+)

1. Performance Testing ‚ö°

Badge: Speed | Efficiency

What: Response time, throughput, resource usage under normal conditions

Measures: Response time, CPU, Memory, Disk I/O, Network bandwidth

Tools: JMeter, LoadRunner, Gatling

Acceptance: Response time < 2 seconds

2. Load Testing üìä

Badge: Heavy Users | Capacity

What: Application under heavy concurrent user load

Example: Simulate 1000 concurrent users accessing banking portal simultaneously

Goal: Find maximum number of users application can handle before degradation

3. Stress Testing üí•

Badge: Breaking Point | Extreme Load

What: Application BEYOND maximum capacity to find breaking point

Scenarios: Double expected load, Hardware failures, Network issues, Database locks

Expected: System handles gracefully OR fails with appropriate error messages (NOT crash)

4. Usability Testing üòä

Badge: User Experience | UI/UX

Questions Tested:

  • Can users complete tasks easily?
  • Is interface clear and intuitive?
  • Are buttons positioned logically?
  • Are colors and fonts professional?

Who: Non-technical users and end-users provide best feedback

5. Compatibility Testing üñ•Ô∏è

Badge: Cross-Platform | Multiple Environments

Operating Systems: Windows, Mac, Linux

Browsers: Chrome, Firefox, Safari, IE, Edge

Devices: Desktop, Mobile, Tablet, Smartwatch

6. Security Testing üîê

Badge: Data Protection | Authorization

Tests:

  • SQL Injection attempts
  • Cross-Site Scripting (XSS)
  • Authentication bypass

7. Reliability Testing üìà

What: Application functions correctly continuously for extended periods

Metric: Mean Time Between Failures (MTBF)

Example: Mobile phone after 10 days continuous use may hang due to RAM buildup

8. Recovery Testing üîÑ

Scenarios:

  • üí• Sudden power failure
  • üåê Network disconnection

9. Authorization & Authentication Testing üîë

Badge: Access Control | User Identity

Authorization: Users have proper access levels (who can do what)

Authentication: System recognizes registered users, provides correct information

Example: Admin ‚Üí all features, Paid User ‚Üí limited features, Free User ‚Üí no features

10. GUI Testing üé®

Badge: Visual Design | Layout

Checks:

  • Alignment of objects (buttons, fields, text)
  • Font sizes, styles, and colors consistent
  • No broken frames or scattered content
  • No object overlapping issues
  • Professional appearance

For:

12. Globalization Testing üåç

I18N (Internationalization):

  • Use property files for each language

Key Takeaways for Testers

  1. Master all testing types - Know when to use each based on context
  2. Combine Functional + Non-Functional - Both needed for complete QA
  3. Early testing saves money - Find defects in requirements phase = 10-100x cheaper
  4. Regression testing is non-negotiable - Every change needs coverage
  5. Prioritize based on risk - Focus testing on critical features first
  6. Document everything - Test cases enable consistency & knowledge transfer
  7. Automate repetition - Manual for new features, automate for regression
  8. Test like an end-user - Ad-hoc and exploratory catch real-world bugs
